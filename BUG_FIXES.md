## Bug 2 的可能原因

根据代码分析，"第一次打开就显示已达上限"可能的原因：

### 原因1: 数据库中有旧数据
如果之前测试过，数据库中可能已经有打开记录。解决方法：
1. 打开应用，进入"设置"页面
2. 点击"查看今日记录"，看看是否有旧数据
3. 点击"清除今日记录"，清除所有数据
4. 重新测试

### 原因2: 限制类型配置错误
如果你使用的是 **TOTAL_ALL（所有应用总次数）** 限制类型：
- 假设你选择了10个应用，限制总次数为10次
- 如果这10个应用之前各被打开过1次，总次数就是10次
- 再打开任何一个应用，就会被阻止

解决方法：
- 使用 **PER_APP（每个应用次数）** 限制类型，这样每个应用可以单独打开10次
- 或者增加 TOTAL_ALL 的限制次数

### 原因3: 限制值设置错误
检查你创建策略时，限制次数是否真的设置为10。可能不小心设置成了0或其他值。

## 查看日志

运行以下命令查看详细日志：

```bash
./debug_logs.sh
```

日志中会显示：
- `getOpenCount`: 当前打开次数
- `Recording app open`: 何时记录打开
- `BLOCKING` 或 `NOT BLOCKING`: 是否阻止
- `Calculated: openCountLimit=X, openCount=Y, remainingCount=Z`: 计算出的限制、当前次数、剩余次数

## 预期行为

如果限制设置为10次（PER_APP模式）：
- 第1-10次打开: 应该成功，显示使用通知，剩余次数递减
- 第11次打开: 应该被阻止，显示对话框

如果第1次打开就被阻止，说明：
1. 数据库中已经有10次或更多的记录（查看今日记录确认）
2. 限制值设置错误（不是10）
3. 限制类型是TOTAL_ALL，且其他应用已经用完了总次数
